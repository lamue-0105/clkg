## 一、我做了什么

按照你技术手册 §6.5 的要求，实现了"采集→入库的桥"。

简单说就是：**采集者用 Excel 模板填好数据 → 你敲一行命令 → 数据自动进入知识图谱数据库**。

这条链路在此次开发之前是不通的——采集模板有了，数据库有了，但中间缺一段能读模板并自动转成数据库能理解的格式的代码。现在这段代码补齐了。

## 二、新建和修改了哪些文件

### 新建了 3 个核心文件

| 文件 | 干什么用 |
|---|---|
| `ingest/collection_schema.py` | 定义了"采集模板到底有哪些列、每列叫什么、必填还是选填、填完对应知识图谱的哪个字段"——是**所有相关代码共享的一份权威定义** |
| `ingest/connectors/template.py` | 实际干活儿的代码：打开 Excel，一行一行读，把填好的数据转成数据库能存的三元组语句 |
| `ingest/pipelines/template.py` | 把上面两步串起来，外加一个可选的语言模型增强（从长文本描述里自动抽地名、人名等） |

### 修改了 2 个已有文件

| 文件 | 改了什么 |
|---|---|
| `ingest/cli.py` | 新增了一条 `template` 命令，这样你就能在终端里敲 `python -m ingest.cli template 文件名.xlsx` 来入库 |
| `data_collection/build_template.py` | 之前这个脚本里自己硬编码了一份列定义。现在改成从 `collection_schema.py` 读取同一份定义——这样改一处，两边自动同步，不会出现模板和入库代码对不上的情况 |
| `data_collection/CLKG_采集模板.xlsx` | 用重构后的脚本重新生成，确认和原来一模一样 |

## 三、设计上做了哪些关键决定

### 决定 1：一份定义，三处共用

`collection_schema.py` 这 320 行代码是整个设计的核心。它把采集模板的 4 张表、90 列、每列的中文名、英文谓词名、是否必填、下拉选什么词表……全部以数据的形式定义好。

这样做的好处是：

- **生成模板的脚本**读它 → 保证了生成的 Excel 表头不会和代码对不上
- **入库的连接器**读它 → 保证了 Excel 里填什么值、数据库里就存到对应的字段
- **未来的验证器**也能读它 → 采集者交数据之前自动检查是否少填、填错

旧方案是三处各自硬编码一份列定义，改一处容易漏改另外两处，导致数据写错地方。

### 决定 2：用"精确匹配"而非"智能解析"

模板表头像 `hasName_名称*` 这种格式。我没有写正则表达式去自动拆解它，而是让 `collection_schema.py` 里把每个表头完整写出来。连接器拿到 Excel 后，用表头字符串去查这份定义。

这比智能解析更笨重，但**绝对不会出错**——表头写错了，就找不到对应的定义，就会报警；表头正确，就精确匹配。不靠猜测。

### 决定 3：跨表关联靠 natural_key

比如 Document 表里有一列 `hasSender_寄件人`，填的是某个人物的 natural_key（如 `qiaopi-ac-chenmiaoqing`）。连接器不负责去数据库里找这个人——它只是原样标注"这是对另一个实体的引用"。等到存储过程 `ingest_batch()` 执行时，数据库会自动查找或创建对应的实体。

这个模式和侨批连接器（`qiaopi.py`）的做法完全一致，保持了代码风格统一。

### 决定 4：GCJ-02/BD-09 坐标暂不转换

国内高德地图用的是 GCJ-02 坐标系，和 GPS 用的 WGS84 有几十到几百米的偏移。正确的做法是写一个坐标转换函数——但这不是最紧急的事。现在的做法是：

- 遇到 WGS84 坐标 → 直接存
- 遇到 GCJ-02 / BD-09 → 标记"需要转换"，先按 WGS84 存进去，打一条警告
- 等后续专门写坐标转换模块时，按标记找到这些记录重新处理

这和你的手册 §10.4 的策略一致："不阻塞入库"。

### 决定 5：示例行三道防线

模板每张表第 2 行是黄色的示例行。入库时必须跳过它，不然会把示例数据也当成真实数据写进数据库。

用了三层检测，从强到弱：

1. **查颜色**：黄色底色（`#FFF2CC`）→ 一定是示例行
2. **查位置**：第 2 行 → 很可能是示例行（按惯例模板始终把示例放在第 2 行）
3. **查内容**：natural_key 等于已知示例值（如 `mustang-pl-lomanthang`）→ 确认是示例行

即使模板被另存后丢失了颜色格式，还有位置和内容兜底。

## 四、处理流程

```
Excel 模板（四张表）
  │
  ├─ 地点表 (27列) → 每行一条"地点"记录，可能带经纬度
  ├─ 文档表 (34列) → 每行一条"文档"记录，含跨表引用（寄件人、寄出地等）
  ├─ 人物表 (12列) → 每行一条"人物"记录
  └─ 资产表 (17列) → 每行一条"照片/录音"记录
  │
  ▼
连接器逐行解析
  · 跳过示例行
  · 跳过空行
  · 红色必填列为空 → 报错
  · 每列查角色定义，分发到对应的数据库字段
  · 经纬度合并为一条空间语句
  │
  ▼
写入临时表（staging）
  │
  ▼
存储过程 ingest_batch()
  · 给每个实体分配永久 ID
  · 解析跨表引用
  · 证据信息统一管理
  │
  ▼
最终存入三张主表
  · evidence（谁在什么时候从什么来源采集的）
  · conceptual_entity（每个遗产对象的注册信息）
  · entity_statement（所有属性、关系、空间坐标）
```

## 六、过程中遇到并修复了哪些问题

### 问题 1：指定了区域但数据写错了库

**现象**：命令行传了 `--region test`（表示写入测试库 `test_cl_kg`），但实际上数据写进了 `mustang_cl_kg`（生产库）。

**原因**：代码里有一段逻辑取了数据行本身的 region 列的值来决策目的地，忽略了用户显式指定的参数。

**修复**：改为以用户指定为准，用户没指定时才从数据中自动判断。

**善后**：误写入 mustang 库的测试数据已经清理干净。

### 问题 2：读取了两次 Excel 文件

**原因**：第一次做 region 探测时读了一次，正式解析时又读了一次——浪费。

**修复**：只在确实需要自动判断 region 时才做探测。

### 问题 3：环境配置文件格式问题

`.env` 文件里有一段文本没有正确注释，导致配置加载时报了一个无害的警告。

**修复**：清理了 `.env` 文件格式。

## 七、现在你能怎么用

### 基础用法

```bash
# 检查数据库状态（以测试库为例）
python -m ingest.cli check test

# 把填好的模板入库到测试库
python -m ingest.cli template 填好的模板.xlsx --region test

# 测少量数据（只处理前 5 行）
python -m ingest.cli template 填好的模板.xlsx --region test --rows 5

# 看看连接器"看到了"什么，但不真的写库
python -m ingest.cli template 填好的模板.xlsx --region test --rows 5 --dry-run
```

### 验证入库是否有问题

在数据库里跑这几条查询：

```sql
-- 看看有哪些类型的实体、各多少个
SELECT entity_type, count(*) FROM conceptual_entity GROUP BY entity_type;

-- 看看有哪些属性、各多少条
SELECT predicate, count(*) FROM entity_statement GROUP BY predicate ORDER BY 2 DESC;

-- 检查跨表引用有没有"断链"（应该为 0）
SELECT count(*) FROM entity_statement
LEFT JOIN conceptual_entity ON object_entity_id = pid
WHERE object_entity_id IS NOT NULL AND pid IS NULL;
```
