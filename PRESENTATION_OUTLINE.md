## 摘要 (Abstract — 1 张幻灯片开篇用)

> 本研究面向**文化景观 (Cultural Landscape)** 这一兼具空间、时间、社会三重属性的研究对象, 针对现有遗产数据基础设施"**异构难、时空难、来源难、语义难**"四类核心瓶颈, 设计并实现了 **CLKG (Cultural Landscape Knowledge Graph)** — 一套基于 PostgreSQL+PostGIS 物化层、融合 W3C 四类核心本体 (RDF-star · 4D-fluents · PROV-O · GeoSPARQL) 的语义底座。两日攻坚以**木斯塘高原考古、华南侨批文献、新疆丝路遗产**三个差异显著的实证场景验证框架普适性, 共构建 ~35,000 实体 / ~480,000 三元语句, 并通过空间推理自动发现 646 条"文化景观单元—子要素"包含关系, 为后续多模态融合与跨域查询提供基础设施。

## 一、研究背景与问题界定 (Slide 1–2)

### 1.1 文化景观数据的"四难"

文化景观研究 (UNESCO 1992 定义) 与单点遗产不同, 强调"**人地互动的累积痕迹**", 数据天然具有以下四类挑战:

| 挑战维度 | 具体表现 | 学科后果 |
|----------|----------|----------|
| **异构性** | 表格 / GIS / 影像 / 文献 / 口述并存 | 单一数据库范式 (关系 / 文档 / 图) 都无法独立承载 |
| **时空性** | 实体随时间演变, 同一名称指代不同位置 | 传统"快照式"建模导致历史信息丢失 |
| **来源性** | 学术结论必须可追溯到原始证据 | 没有 provenance 的 KG 在人文学科不可用 |
| **跨域语义** | 不同地区 / 学者 / 时期使用不同术语体系 | 字符串匹配无法支撑跨域综合研究 |

## 三、方法论 (Slide 4–5)

### 3.1 CL-Onto 三表归一模型 (核心创新)

```
┌────────────────────────────────────────────────────────────────────┐
│                  CL-Onto 三表归一模型                                │
├─────────────────────┬────────────────────────────────────────────┤
│ 表                  │ 语义角色                                     │
├─────────────────────┼────────────────────────────────────────────┤
│ evidence            │ PROV-O 溯源 — 每条事实回溯到原始数据源       │
│                     │ (source_type, source_name, collector,        │
│                     │  recording_date, metadata)                   │
├─────────────────────┼────────────────────────────────────────────┤
│ conceptual_entity   │ 实体注册表 — 全局唯一 PID, CL-Onto 类型体系   │
│                     │ (pid PK · entity_type · label_zh / label_en) │
├─────────────────────┼────────────────────────────────────────────┤
│ entity_statement    │ RDF-star + 4D-fluents 语句承载层            │
│                     │ (subject · predicate ·                       │
│                     │  object_value / object_entity / geometry ·   │
│                     │  valid_time_start / end · evidence_id)       │
└─────────────────────┴────────────────────────────────────────────┘
```

**实体类型体系 (CL-Onto Type Taxonomy)**:
- `pl` — Place (原子地点)
- `clu` — **CulturalLandscapeUnit (文化景观单元 — 本研究核心概念)**
- `ac` — Actor (人物 / 机构)
- `ev` — Event (事件)
- `img` — DigitalImage (数字影像)
- `doc` — Document (文献)

**PID 命名空间设计**:
```
clkg:{region}:{type_abbr}-{temporal_info}-{7-digit-serial}

例: clkg:mustang:clu-15c-0000023        ← 木斯塘 15 世纪文化景观单元 #23
    clkg:qiaopi:doc-19c-0001847         ← 侨批 19 世纪文献 #1847
    clkg:xinjiang:pl-han-0002104        ← 新疆汉代地点 #2104
```

> 双层命名空间 (`region` × `type`) 既保证**区域独立性** (各库可独立部署), 又留出**未来联合查询的语义钩子**。

### 3.3 关键设计原则

| 原则 | 含义 | 实现 |
|------|------|------|
| **Staging 单一入口** | Python 只写 staging, 不直接动业务表 | `ingest_batch()` 是唯一事务边界 |
| **幂等性 (Idempotency)** | 同 batch_id 重跑不产生重复 | `ON CONFLICT ON CONSTRAINT unique_stmt_check DO NOTHING` |
| **追溯完整性 (Traceability)** | 每条语句必有 `evidence_id` | FK 约束强制, 无可逃逸路径 |
| **空间一致性 (Spatial Coherence)** | 所有几何统一 SRID 4326 | `ST_Transform` 在摄取层执行 |
| **类型可扩展 (Extensibility)** | 新增 (region, type) 不需建库 | PID 序列**懒创建** (lazy minting) |

### 4.3 典型发现 (现场演示)

#### Finding 1: 文化景观单元的自动浮现 — Lomanthang 案例

**方法**: 不预先标注哪些是 CLU, 而让 `ST_Contains(polygon, point)` 自动从空间关系**反向推导语义包含关系**。

**结果**:
```
LOMANTHANG (古城) 自动获得 68 个子实体
TSARANG (聚落)    自动获得 34 个子实体
GHEMI            自动获得 22 个子实体
... 合计 646 条 containsPlace 三元组
```

**意义**: "**文化景观单元**" 这一抽象概念**从几何关系中自动浮现**, 而非人工先验给定 — 是对 UNESCO 文化景观定义"作为整体被感知"的 KG 实现。

#### Finding 2: 跨学科语义可达 — 三个"世界级"考古点

**方法**: 在 5409 个新疆文保单位中, 用单条 `WHERE predicate='hasHeritageLevel' AND object_value LIKE '%世界级%'` 查询。

**结果**: **高昌故城 (高昌区) / 北庭故城 (吉木萨尔县) / 雅尔湖故城 (高昌区)** — 三处均为 "丝绸之路: 长安—天山廊道路网" 申遗组成部分, 全部位于吐鲁番盆地的丝路咽喉地带, 年代均跨越公元 500–640 年 (麴氏高昌 → 唐西州时期)。

**意义**: KG 从 5409 条记录中**准确识别**最高级别遗产, 等于自动验证申遗体系。这是**字符串属性**到**学术结论**的语义距离压缩。

#### Finding 3: 田野研究行为的反向重建

**方法**: 6036 张照片 EXIF → `capturedAt` + `locatedAt` → 按月聚合。

**结果**: 可还原多年田野考察的时空轨迹, 实现 PROV-O 提倡的"**研究过程本身**也是知识对象"的范式。

## 五、贡献与讨论 (Slide 8)

### 5.1 理论贡献 (Theoretical Contributions)

1. **CL-Onto**: 首次将"**文化景观单元 (CLU)**"作为一等本体类型纳入遗产 KG, 区别于传统的原子 Place, 填补 UNESCO 文化景观概念在 KG 层的实现空白。
2. **三表归一模型**: 证明 W3C 四类核心本体 (RDF-star · 4D-fluents · PROV-O · GeoSPARQL) 可在**单一物化关系层**中以**三张表**同时实现, 兼顾**语义表达力**与**查询性能**。
3. **空间—语义双向耦合**: 提出"几何关系→语义关系"的自动浮现机制 (ST_Contains → containsPlace), 是 GIScience 与知识工程的具体融合点。

### 5.2 方法学贡献 (Methodological)

1. **反熵 Schema 治理三件套** (dump_schema · init · verify): 解决多区域并行部署的 schema drift 顽疾。
2. **PID 双层懒创建命名空间**: 兼顾区域独立与未来联合, 任何新 (region, type) 组合无需 DBA 介入。
3. **连接器—管道两层架构**: 数据源差异收敛在 connectors 层, 业务逻辑在 pipelines 层 — 加新数据源 = 写 ~100 行 Python。

### 5.3 应用贡献 (Applied)

- 三个示范区域可直接服务相关项目研究问题
- 框架代码可被同实验室师弟师妹复用 (建库半小时, 加新模态 ~1 天)
- 后续可对接 RDF 导出 → CIDOC-CRM / CHGIS 国际标准

### 5.4 局限性 (Limitations)

## Q&A 预测 + 应答提纲

**Q1: CL-Onto 与 CIDOC-CRM 是什么关系?**
> A: CIDOC-CRM 是事件中心 (Event-centric) 的博物馆/藏品本体, 重点描述 "什么人在什么时间对什么物做了什么"。CL-Onto 是**景观中心** (Landscape-centric), 重点是"什么实体在什么时空范围内表现出什么属性"。两者**互补**: CL-Onto 实例可映射到 CRM 的 E53_Place / E5_Event / E22_Human-Made_Object — 这是后续 RDF 导出的对接点。

**Q2: 为什么不用 Neo4j / GraphDB 直接做 RDF triple store?**
> A: 三个理由 — (1) **空间能力**: PostGIS 的 GeoSPARQL 实现比任何 triple store 都成熟; (2) **学术界 SQL 素养**: 让研究者继续用熟悉的 SQL, 而不是 SPARQL; (3) **物化即可观测**: 关系表可被 pgAdmin 直接审计, 利于本体迭代。Triple store 留给最终发布层 (导出 Turtle/JSON-LD)。

**Q3: 在三个区域上验证够吗? 怎么证明框架普适?**
> A: 这是**域内普适性 (within-domain) 的初步验证**, 不是普适证明。三个区域的**异构维度**已经覆盖空间尺度 (50km² ↔ 跨国)、时间尺度 (中世纪 ↔ 史前~清)、模态 (表格/GIS/影像/文献) 的差异。下阶段会做 between-domain 联合查询 (如"侨批寄出地"是否落在"丝路廊道遗产"附近) 作为更强的普适检验。

**Q4: 没有 LLM 这套还有什么意义?**
> A: LLM 是**催化剂**, 不是**结构**。当前 35,000 实体 / 48 万语句**全部来自规则化抽取**, 完全可控、可解释、可审计 — 这恰恰是人文学科最需要的特性。LLM 上来后处理的是长文本暗藏实体, 是**锦上添花**, 而不是**底层支撑**。

**Q5: 这个框架能给师弟师妹用吗?**
> A: 这正是设计目标之一。`init <region>` 一行命令建新库; 新数据源 = 写 ~100 行 Python connector; CL-Onto 本体文档化在 `00_create_schema.sql` 中。**目的不是只让我一个人能跑, 而是让框架成为团队基础设施**。
