================================================================================
第 1 章  研究背景与问题界定
================================================================================

1.1 研究对象的特殊性
  文化景观(Cultural Landscape, UNESCO 1992)区别于单点遗产, 强调"人地互动的累积
  痕迹", 数据天然兼具空间、时间、社会三重属性, 并具有持续累积、多源汇入的特征。
  这决定了其知识组织不能停留在"藏品登记", 而须支持时空演变与跨源综合。

1.2 文化遗产数据的"四难"(核心瓶颈)
  D1 异构难: 表格 / GIS / 影像 / 文献 / 口述并存; 单一数据库范式(关系/文档/图)均
            无法独立承载。
  D2 时空难: 实体随时间演变, 同名异指、同指异名; 快照式建模丢失历史信息。
  D3 来源难: 人文学术结论必须可追溯到原始证据; 无 provenance 的知识图谱在人文学科
            不可用。
  D4 语义难: 不同地区/学者/时期使用不同术语体系; 字符串匹配无法支撑跨域综合研究。

1.3 现有范式的局限(三类)
  P1 传统 GIS(空间优先, ArcGIS/QGIS): "图层+属性表", 同一实体跨图层被肢解, 缺语义层。
  P2 元数据/本体标准(语义优先, Dublin Core / CIDOC-CRM): 偏文档/藏品级, 几何推理
     薄弱, 缺时间维与语句级溯源。
  P3 通用知识图谱(规模优先, Wikidata/DBpedia): 颗粒度过粗, 学者无法精确引用单条
     事实的证据来源。

1.4 代表文献与研究空缺
  代表性工作(GIS×本体、CIDOC-CRM 系、NLP-KG、区域非遗 KG 等)多在单一维度有突破,
  但无一能同时承载"空间几何 + 时间区间 + 来源溯源 + 跨域语义对齐"四维; 多模态融合
  普遍空白。本研究直面这一空缺。
  (代表文献清单见 PRESENTATION_OUTLINE.md §1.3, 正式投稿时按目标期刊格式重排。)

  【★绘图 F1｜研究问题树: 四难 → 现有范式局限 → 研究空缺｜

================================================================================
第 2 章  研究目标、研究问题与假设
================================================================================

2.1 总目标
  为多源异构的文化遗产/文化景观数据, 构建一个统一、可扩展、可追溯的语义底座, 同时
  支持空间几何推理、时间区间表达、来源证据链与跨区域语义对齐; 并提供一套可被团队
  规模化复用的、从一手采集到入库验证的完整方法流程。

2.2 核心研究问题(RQ)与子问题
  RQ: 如何为多源异构的文化景观数据构建一个统一、可扩展、可追溯的语义底座, 同时支持
      空间几何推理、时间区间表达、来源证据链与跨区域语义对齐?
  SQ1 异构摄取: 如何用统一数据模型 + 可插拔连接器承载多模态数据源?
  SQ2 时空表达: 如何在关系存储中同时表达几何(GeoSPARQL)与时间区间(4D-fluents)?
  SQ3 来源溯源: 如何使每条事实强制可回溯到原始证据(PROV-O)?
  SQ4 语义互操作: 如何用统一类型体系与命名空间支持跨区域语义对齐?
  SQ5 规模化生产(本阶段新增): 如何组织多名采集者产出"进库即规整"的一手数据, 使来源
      溯源与语义一致性在采集源头即被满足, 而非事后修补?

  【★绘图 F2｜RQ→SQ 分解图｜顶部 RQ, 向下分出 SQ1–SQ5, 每个 SQ 下挂其技术承载
   (连接器矩阵 / 4D-fluents+GeoSPARQL / PROV-O 证据表 / CL-Onto 类型体系 / Phase 0
   采集协议)。SQ5 用不同色块标出"本阶段新增"。】

2.3 研究假设(供实验设计检验, 见第 7 章)
  H1 域内普适/可承载: 同一套 CL-Onto 模型与同一摄取流水线, 能承载异构维度(空间尺度/
     时间跨度/数据模态)显著不同的多区域数据, 且来源、时空、几何信息完整保留、无逃逸路径。
  H2 空间—语义浮现: 文化景观单元(CLU)与其子要素的"包含"语义, 可由几何关系
     (ST_Contains)自动推导, 而非人工先验标注。
  H3 前置式采集的质量增益: 受控词表 + 强制来源 + 统一 natural_key 的前置式采集协议
     (Phase 0), 相比"自由采集 + 事后归一化", 能显著降低谓词值域离散度与人工归一化
     工作量, 提升来源完整率, 并降低单位记录的采集+清洗总工时。
  H4 跨域可联通(后续): 双层命名空间(region × type)留出的语义钩子, 能支撑
     between-domain 联合查询(如"侨批寄出地"与"丝路廊道遗产"的空间邻近分析)。

第 3 章  理论框架与本体设计

3.1 三学科支柱、本体架构与标准合规

  (1) 三学科支柱与融合
  · GIScience(空间认知): GeoSPARQL、ST_Contains 等空间谓词。
  · 知识工程(本体与知识表示): RDF-star(语句级注记)、4D-fluents(时间区间)、PROV-O(溯源)。
  · 遗产学(文化景观): UNESCO 1992 定义、CIDOC-CRM 概念框架。
  三者融合落地为本研究领域本体 CL-Onto。理论贡献: 在一个物化关系存储中同时实现 W3C
  四类核心本体语义, 区别于纯 RDF triple store(性能差)与纯关系表(语义浅)两个极端。

  【★绘图 F3｜三学科支柱与本体融合图｜三支柱(GIScience/知识工程/遗产学)各列其标准
   (GeoSPARQL/RDF-star+4D-fluents+PROV-O/UNESCO+CIDOC-CRM), 向下汇聚到 CL-Onto,
   再落到 PostgreSQL+PostGIS 物化层。复用 PRESENTATION_OUTLINE §2.2 草图升级为正式图。】

  (2) 本体架构: OWL → T-Box → A-Box 与四标准外挂
  以语义网标准栈表述, CL-Onto 的构造分三层, 并向 A-Box 外挂四项标准:
  · OWL(本体建模): 声明类、属性与约束, 是 CL-Onto 的形式定义语言。
  · T-Box(本体骨架): 术语层 = 实体类型体系(§3.3) + 谓词集(§5.3) + 约束(FK /
    unique_stmt_check / 几何 SRID 4326)。回答"有哪些类与关系、受什么约束"。
  · A-Box(实例数据): 断言层 = 三表中的实例与语句(conceptual_entity + entity_statement
    + evidence)。回答"现实世界的具体事实"。本研究的"三表归一模型"(§3.2)正是 A-Box
    在物化关系层的实现。
  四项外挂标准, 各司其职:
  · URI   唯一标识      → PID 命名空间(§3.4)。
  · SKOS  多语言/异名   → label_zh/label_en + 别名(跨语言对齐, 见 §9.2 局限)。
  · WKT   空间表达      → stmt_value.wkt → object_geometry(4326)(§5.4)。
  · SHACL 规则校验      → 约束与"形状"校验; 本研究将其前移到采集端(Phase 0 验证器, §6.5)。

  【★绘图 F16｜本体架构 OWL→T-Box→A-Box 与四标准外挂图｜顶 OWL(本体建模)—定义类/属性/
   约束→ T-Box(本体骨架)—实例化→ A-Box(实例数据=三表); A-Box 四向外挂 URI/SKOS/WKT/
   SHACL, 每标准旁以色标注 已实现/部分/缺口。第 3 章框架主图(对应用户提供的标准栈图)。】

  (3) 标准合规现状与缺口(对照 F16)
    标准    对应 CLKG 实现                              现状
    OWL    CL-Onto 形式定义                            缺口: 未用 OWL 形式化, 本体散在
                                                        SQL 注释 + 连接器代码
    T-Box  类型体系 + 谓词 + 约束                      隐式: 物化在关系 schema/代码,
                                                        非声明式 OWL/RDFS
    A-Box  三表实例(entity/statement/evidence)          已实现: 最成熟的一层
    URI    PID(clkg:region:type-temporal-serial)       已实现
    SKOS   label_zh/label_en + 别名                    弱: label_en 多为空, 跨语言对齐
                                                        仅 Mustang 完整
    WKT    stmt_value.wkt → object_geometry(4326)       已实现
    SHACL  DB 约束 + Phase 0 采集验证器                  部分: 有 FK/unique/SRID, 尚未
                                                        形式化为 SHACL shapes
  关键判断:
  · CLKG 现状是"强 A-Box、隐式 T-Box"——实例数据扎实, 但本体骨架尚未声明式形式化。
    形式化 T-Box(OWL/RDFS)是标准合规与对接 CIDOC-CRM 的关键一步(路线见 §10.6)。
  · SHACL ↔ Phase 0 验证器是同一机制的两种表述: 本研究把"形状校验"前移到采集那一刻,
    给 H3 前置式采集一个标准学名 —— provenance-at-source + SHACL-at-collection。
  · SKOS 是补齐多语言/异名缺口的标准路径(接 Wikidata/GeoNames, 路线见 §10.6)。

3.2 CL-Onto 三表归一模型(核心创新; = A-Box 的物化实现)
  本节描述 §3.1 中的 A-Box 如何在 PostgreSQL 中以三张表落地。
  表 1 evidence —— PROV-O 溯源层
    语义: 每条事实回溯到原始数据源(谁/何时/从何处/如何采集)。
    关键列: id(PK) · source_type · source_name(URI/引用键) · collector · recording_date
            · description · metadata(jsonb)。
    去重: 同 source_name 的证据在批内复用一行(多语句共享一条证据)。
  表 2 conceptual_entity —— 实体注册层
    语义: 全局唯一持久标识 PID + CL-Onto 类型。
    关键列: pid(PK,varchar) · entity_type(NOT NULL) · label_zh · label_en · created_at。
    label_zh 为从首条 hasName 派生的只读便捷字段, 真值仍在语句层。
  表 3 entity_statement —— RDF-star + 4D-fluents 语句承载层
    语义: 一行 = 一条 4D-fluent 三元语句, 可挂时间区间与证据、置信度。
    关键列: statement_id(PK) · subject_id(FK→entity,NOT NULL) · predicate(NOT NULL)
            · object_value(字面量) / object_entity_id(FK→entity) /
              object_geometry(geometry,4326) · valid_time_start / valid_time_end(text)
            · evidence_id(FK→evidence) · confidence(double,默认 1.0)。
    宾语三槽互斥: 字面量、实体引用、几何三选一, 分别承载属性、关系、空间事实。
    去重约束: UNIQUE(subject_id,predicate,object_value,object_entity_id,
            valid_time_start,evidence_id) —— 幂等重跑的基础。
  属性建模: 不用宽列, 而以"一属性一语句"的 SPO 形式存入 entity_statement; 同一实体类型
    可拥有完全不同的谓词集合(Mustang 的 hasLayer vs 侨批的 hasCurrency), 无需 schema 迁移。

  【★绘图 F4｜CL-Onto 三表归一模型 ER 图｜三表与外键关系(entity_statement.subject_id /
   object_entity_id → conceptual_entity.pid; entity_statement.evidence_id → evidence.id),
   标注三槽宾语、valid_time、confidence、unique 约束。论文 Methods 主图之一。】

3.3 实体类型体系(CL-Onto Type Taxonomy)
  pl  Place               原子地点(可单独定位的一处)
  clu CulturalLandscapeUnit 文化景观单元(群体/区域等聚合实体 —— 本研究核心概念)
  ac  Actor               人物 / 家族 / 机构
  ev  Event               事件(预留)
  img DigitalImage        数字影像 / 多模态资产
  doc Document            文献 / 档案 / 口述史转写
  clu vs pl 判定: 一群同类对象或一片区域 → clu; 可单独定位的一处 → pl。

3.4 PID 持久标识命名空间与三库关系
  格式: clkg:{region}:{type_abbr}-{temporal_info}-{7位序列号}
  示例: clkg:mustang:clu-15c-0000023 / clkg:qiaopi:doc-19c-0001847 /
        clkg:xinjiang:pl-han-0002104
  设计: 双层命名空间(region × type)既保证区域独立部署(per-region database isolation),
        又为未来联合查询留语义钩子; 序列懒创建, 新 (region,type) 无需建库迁移。
  三库关系(现状): mustang / qiaopi / xinjiang 三库物理隔离、各自独立; 仅通过统一的
        CL-Onto 本体、统一 PID 命名规则、统一摄取契约在"方法层"耦合; between-domain
        联合查询(H4)为规划项, 尚未落地。

  【★绘图 F13｜CLKG 三库关系结构图｜(待用户提供飞书原图后据其修订)。应表达三库的
   物理隔离 + 方法层共享(同一 CL-Onto / 同一 PID 规则 / 同一 ingest 契约) + 规划中的
   联合查询钩子。若用户原图含我未掌握的关联设计, 以原图为准更新本节与 §7.3、§7.8。】

3.5 标识的双键设计(natural_key vs PID)
  · natural_key(业务键): 人在采集时给定、有意义、可复现(如 mustang-pl-lomanthang),
    用于指认现实世界的同一对象, 是去重锚点。采集者只接触它。
  · PID(代理键): 系统自动发号、不透明、永久不变, 作为主键与所有外键引用。
  · 绑定: pid_registry(PK: region,type_abbr,natural_key) 存映射;
    resolve_or_mint_pid() 保证"同 natural_key 永远解析到同一 PID"(幂等), 首见即发号、
    后见即命中, 从而天然去重。
  · 失配后果: 同一对象被填成两个 natural_key → 两个 PID → "双胞胎"实体, 须人工归并。
    故采集前必须查权威清单(见 §6.2)。

  【★绘图 F5｜natural_key→PID 解析流程图｜输入 natural_key → 查 pid_registry →
   命中(返回旧 PID, 不新建) / 未命中(mint_pid 发新号 + 写 registry) → 返回 PID。
   标出"幂等去重锚点"。用于解释方法的实体一致性机制。】

第 4 章  系统架构与存储层
================================================================================

4.1 总体架构(自底向上)
  存储层: PostgreSQL + PostGIS, 按区域独立建库, 库名约定 {region}_cl_kg。
  治理层: staging schema + PID 发号器 + ingest_batch() 存储过程 + 权限(4 个 SQL 文件)。
  摄取层: Python ingest 包(connectors → staging → ingest_batch → 三表)。
  采集层(Phase 0, 新增): 采集模板 + 验证器 + 云收件口 + NAS 主存(第 6 章)。
  语义浮现层: NER(Qwen, 可选)+ 空间推理(ST_Contains)。
  审计层: 跨区域/跨谓词分布审计, 反哺本体(Phase 4)。

  【★绘图 F6｜系统总体分层架构图｜自底向上六层(存储/治理/摄取/采集/语义浮现/审计),
   右侧标注各层关键产物与不变量(单一入口/幂等/追溯/空间一致/类型可扩展)。系统总览主图。】

4.2 区域独立 + 反熵 Schema 治理
  每个区域库按序部署: 01_staging.sql → 02_pid_minter.sql → 03_ingest_batch.sql →
  04_grants.sql。反熵三件套(dump_schema / init / verify)防止多区域并行部署的
  schema drift。03_ingest_batch.sql 顶部以注释固化对业务表列名的假设, 列名变更须同步。

4.3 物理数据布局(NAS 约定)
  {NAS_ROOT}/{region}/tabular/*.csv,*.xlsx
  {NAS_ROOT}/{region}/photos/{natural_key}/*.jpg  (文件夹名=地点 natural_key, 自动关联)
  {NAS_ROOT}/{region}/scans|audio/...             (扫描件/录音)
  {NAS_ROOT}/{region}/gis/*.shp,*.geojson         (矢量/栅格)
  原则: 数据库只存路径(file://...), 不存二进制; NAS_ROOT 默认 /Volumes/clkg_data, 由 .env 配置。

4.4 关键设计原则(系统不变量)
  · Staging 单一入口: Python 只写 staging; ingest_batch() 是唯一事务边界。
  · 幂等性: 同 batch_id 重跑不产生重复(ON CONFLICT ON CONSTRAINT unique_stmt_check)。
  · 追溯完整性: 每条语句必有 evidence_id, FK 强制, 无逃逸路径(支撑 SQ3/D3)。
  · 空间一致性: 所有几何统一 SRID 4326, ST_Transform 在摄取层执行(支撑 SQ2/D2)。
  · 类型可扩展: 新增 (region,type) 无需建库, PID 序列懒创建(支撑 SQ4)。

第 5 章  摄取流水线与四阶段方法论闭环

5.1 四阶段闭环 + Phase 0

5.4 坐标系处理(方法学要点, 易错)
  原则: 采集与连接器层如实记录原始坐标 + 声明原始 CRS; 统一在摄取层 ST_Transform 到
  4326, 绝不在采集端手工换算。已知坑与对策:
  · Mustang CSV 为 UTM 44N(EPSG:32644, 非 45N), 连接器显式指定 SRID。
  · 侨批坐标为 GCJ-02(高德), 与 WGS84 偏移 50–500m; GCJ-02/BD-09 非普通 EPSG 代码,
    需先做基准面转换再赋 SRID(待在模板连接器实现)。
  CRS→SRID: WGS84→4326, CGCS2000→4490, EPSG:32644/32645 原样; GCJ-02/BD-09→需基准面
  转换; unk→空。

第 6 章  Phase 0 —— 一手数据采集与协作生产方法(本阶段核心新增)
================================================================================

6.1 定位与方法论主张
  Phase 0 位于 Phase 2 上游, 解决"多名采集者如何产出进库即规整的一手数据"。核心主张:
  把异构归一(D1)、来源溯源(D3)、语义一致(D4)三类约束前移到采集那一刻, 而非入库后用
  Phase 4 审计修补 —— 即"采集即约束 / provenance-at-source"。该主张即假设 H3 的检验对象。

6.2 采集协议(SOP)八要素
  ① 采集单元: 一行 = 一个实体; 跨实体关系靠 natural_key 引用。
  ② 来源强制: 每行必填 记录人/采集日期/source_name/source_type; 文献优先填 Zotero
     citation key, 档案填馆藏号, 网络填 URL, 田野填调查表号; 置信度按亲见/二手/推断分级。
  ③ natural_key 一致性: 同一对象永远同一 key; 优先复用既有稳定编号; 采集前查权威清单。
  ④ 空间标准: 如实记录坐标 + 声明 CRS, 不自行换算(见 §5.4)。
  ⑤ 多模态资产: 按 NAS 目录约定存放, 照片置于以 natural_key 命名的文件夹自动关联;
     库内只存路径; 标注版权/许可。
  ⑥ 去重/认领: 维护权威实体清单, 采集前认领与查重, 避免双胞胎实体。
  ⑦ 采集模板: 受控词表下拉 + 必填校验 + 金标准示例行。
  ⑧ 质量闸门: 采集者自查 → 审核员互查/抽查 → 入库 → Phase 4 审计。

6.3 采集模板(人机界面)
  CLKG_采集模板.xlsx: 四张采集表(Place_地点 / Document_文档 / Actor_人物 / Asset_资产)
  + 词表。采集者填"宽表"(一行一实体, 列名对齐谓词如 hasName_名称), 由连接器在摄取
  阶段 melt 为 SPO 三元组。受控词表覆盖 entity_type / region / source_type / CRS /
  layer / heritage_level / actor_role。

6.4 采集物理流: NAS 主存 + 云收件口(本阶段选定方案)
  分三层各按其优化目标选址:
  · 二进制资产(GB–TB, 持续增长): NAS 主存(成本/容量/数据主权) + 云端冷备份(耐久性底线)。
  · 协作收件口: 采集者上传到"同步到本地文件夹"的云盘(坚果云/Nextcloud/对象存储均可),
    本机据此拉回 NAS 并入库; 不直接暴露 NAS 公网。
  · 数据库: 暂留本地; 当多人需实时查询 / 7×24 自动入库 / 对外 demo 时再迁云。
  服务无关设计: 收件流水线只认本地同步文件夹, 不绑定具体云服务。

  【★绘图 F9｜Phase 0 采集物理流图｜采集者(填模板+资产)→云收件口(同步文件夹)→本机
   拉取→验证器(质量闸门)→NAS 主存(资产)+模板连接器→staging→三表; 旁挂云端冷备份。
   标出"服务无关"与"不暴露 NAS 公网"。】

6.5 采集→入库的桥(模板连接器 + 验证器, 待实现)
  · connectors/template.py: 读标准模板四表, 按列名→谓词映射 emit StatementRow, 完成
    CRS→SRID(含 GCJ-02/BD-09 基准面转换)与 melt。
  · 验证器(收件时执行 §6.2-⑧): 必填校验、词表合法性、坐标-CRS 配套、natural_key 去重
    与权威清单比对、示例行残留检测; 不通过则隔离并产出原因报告。
  · 单一事实源: ingest/collection_schema.py 同时供模板生成、验证器、连接器使用, 防三者漂移。
  · 验证器的统计输出(归一化前离散度、来源完整率、冲突率等)直接作为 H3 实验的测量装置。

7.7 H3 专项实验(前置式采集的质量增益 —— 本阶段方法学重点)
  研究问题: 前置式采集协议相对"自由采集 + 事后归一化", 在数据质量与总成本上有何差异?
  设计: 同一批待采资料, 随机分派或交叉设计为两组——
    对照组 B(基线): 自由格式采集(无受控词表/无强制来源/自拟 key) → 入库 → 事后审计归一化。
    实验组 T(前置式): 用 Phase 0 模板 + 验证器采集 → 入库。
  控制变量: 采集者经验(建议同人先后做 B/T 或拉丁方交叉, 平衡学习效应)、资料难度、单位
    工作量定义。记录采集者元信息。
  测量指标:
    M1 谓词值域离散度: 每谓词 distinct(object_value) 数 / 该谓词语句数(归一化前)。
    M2 事后归一化率: 入库后仍需 UPDATE 修正的条目 / 总条目。
    M3 来源完整率: 含可定位 source_name 的记录占比。
    M4 natural_key 冲突率(双胞胎率): 经人工判定指向同一对象却产生多 PID 的比例。
    M5 单位记录总工时: 采集 + 清洗合计(分别计时)。
    M6(可选)下游可用性: 同一组查询在 T/B 数据上的正确返回率。
  统计处理: 配对/独立样本检验(视设计), 报告效应量与置信区间; 预注册指标口径与样本量。
  预期: T 在 M1/M2/M4 显著低于 B, M3 显著高, M5 总工时更低(清洗成本前置且摊薄)。
  意义: 为"前置式一手采集协议"作为方法学贡献提供量化证据(支撑论文方法学主张)。

第 9 章  贡献、局限与效度威胁

9.1 贡献
  理论: (1) 首次将"文化景观单元 CLU"作为一等本体类型纳入遗产 KG;
        (2) 三表归一模型在单一物化关系层同时实现 W3C 四类核心本体;
        (3) 几何关系→语义关系的自动浮现机制(ST_Contains→containsPlace)。
  方法学: (1) 反熵 Schema 治理三件套; (2) PID 双层懒创建命名空间; (3) 连接器—管道两层
        架构; (4) 本阶段新增: 前置式一手采集协议(Phase 0, provenance-at-source), 并以
        H3 对照实验提供量化证据。
  应用: 三示范区域可直接服务研究; 框架可被团队复用(建库半小时, 加新模态约 1 天);
        可对接 RDF 导出 → CIDOC-CRM/CHGIS。

9.2 局限
  · LLM-NER 尚未全面启用, 长文本暗藏实体未抽取。
  · 时间表达仍为字符串('15c','unk'), 非 ISO/OWL-Time 区间。
  · 跨语言对齐(label_zh↔label_en)目前仅 Mustang 完整。
  · 当前主要 within-domain 验证, between-domain 联合查询(H4)未展开。
  · GCJ-02/BD-09 基准面转换待在模板连接器实现。

9.3 效度威胁(Threats to Validity)
  · 构念效度: "普适性"以三区域近似而非全域证明; 准确率等依赖人工抽检, 有标注主观性。
  · 内部效度(H3): 前置式 vs 基线须控制采集者经验差异(交叉设计/同人先后)与学习效应。
  · 外部效度: 三区域均为中/南亚与中国语境, 推广到其他文明语境仍需检验。
  · 结论效度: 规模数为约数, 须冻结快照、固定版本后精确统计并报告效应量。
